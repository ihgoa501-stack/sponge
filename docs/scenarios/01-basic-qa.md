# 场景 1：基本问答

> 最简单的 Sponge 使用场景——问一个问题，得到一个流式回答，看到成本报告。

---

## 命令

```bash
sponge run "解释 CAP 定理，用一句话"
```

## 期望输出

### 流式阶段（字符逐个出现）

```
CAP 定理指出，在分布式系统中，一致性（Consistency）、可用性（Availability）和分区容错性（Partition Tolerance）三者不可兼得，最多只能同时满足两个。
```

### 收尾（session 结束后）

```

  ─────────────────────────────────
  Savings vs naive:     $0.0072  (36.7%)

```

## 发生了什么

| 步骤 | 说明 |
|------|------|
| 1. 缓存检查 | SHA256(任务 + system prompt) → 未命中（第一次运行） |
| 2. 插件路由 | 没有匹配的插件 → 需要 LLM |
| 3. 成本估算 | tiktoken 扫描 → 估算 $0.0042（输入）+ ~$0.008（输出） |
| 4. 断路器检查 | $0.0124 < $2.00 → 通过 |
| 5. 上下文压缩 | 只有 1 轮对话，没有可压缩的内容 → 穿透 |
| 6. 流式调用 | Opus 4.7 开始生成 → 字符逐个显示 |
| 7. 成本记录 | UsageEvent → $0.0124 |
| 8. 缓存写入 | 缓存结果供将来使用 |

## 关键观察

- **首次运行成本最高**——缓存 miss，系统 prompt 也 miss
- **第二次运行相同问题** → 精确匹配缓存命中 → $0.00
- **第二次运行类似问题** → 语义缓存命中（cosine ≥ 0.95）→ ~$0.0001
- **36.7% 节省仅来自 prompt cache**（system prompt 的缓存读折扣）
