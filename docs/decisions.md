# Architecture Decision Records

> ADR 格式记录 Sponge 的每个关键设计选择——**背景、选项、决策、后果**。
> 让未来的开发者和 agent 理解为什么系统是今天这个样子。

---

## ADR-001: 最终推理模型不静默降级

**状态：** 已采纳 · **日期：** 2026-05

### 背景

现有 AI agent 工具（Copilot HyDRA、Cursor 手动切换、Claude Code `/effort`）的通用省钱策略是：简单任务用便宜模型，复杂任务用贵模型。这意味着用户每次使用都是在**质量和价格之间做权衡**。

### 考虑的方案

| 方案 | 描述 |
|------|------|
| **A: 模型路由（行业标准）** | 根据任务复杂度自动选择模型。简单问题 → Sonnet/DeepSeek，复杂问题 → Opus。 |
| **B: 用户手动切换** | 用户自己决定每个任务用什么模型。CLI 参数 `--model`。 |
| **C: 稳定最终模型 + 基础设施压缩（选定）** | 最终推理模型由用户配置。成本节省来自缓存、压缩、插件路由和辅助执行器，不是静默替换最终模型。 |

### 决策

选择 **C**。核心约束：**"The final reasoning model is stable unless the user changes it."** 如果允许静默降级，缓存系统不够好也没关系——"反正可以路由到便宜的"。稳定最终模型意味着每一层都必须真的减少 paid-token footprint，且辅助执行器的输出必须可审计。

### 后果

**正面：**
- 回答质量边界清晰——最终推理模型不会被系统偷偷换掉
- 架构纪律——所有优化都是真的优化，不是借口
- 差异化定位——把成本节省放在上下文、缓存、工具和审计层

**负面：**
- 冷启动成本更高——在缓存和压缩收敛之前，前几个 session 比路由方案贵
- 边缘任务也可能付高价——即使只是"Hello world"也默认走配置的最终模型
- 用户需要显式配置模型变更——不能依赖系统自动路由来绕过某个 provider 的问题

## ADR-002: 流式响应作为默认

**状态：** 已采纳 · **日期：** 2026-05

### 背景

原始设计是阻塞式 `llm.chat()`——发送消息，等待完整响应，然后一次性显示。这在概念上简单，但用户体验差：30 秒的空白期后突然出现一堵文字墙。

### 考虑的方案

| 方案 | 描述 |
|------|------|
| **A: 阻塞（原始设计）** | `response = llm.chat(messages)` → 等待 → 一次性打印 |
| **B: 流式（选定）** | `async for event in llm.stream(messages)` → 实时渲染 |
| **C: 混合** | 短请求阻塞，长请求流式——根据预估输出长度动态选择 |

### 决策

选择 **B**。理由：

1. **感知性能 = 实际性能**——用户愿意等 30 秒的流式输出，但不愿意等 5 秒的空白
2. **所有主流 provider 都支持流式**——Anthropic、OpenAI、DeepSeek
3. **增量成本追踪**——流式可以在输出产生时就统计 token，而不是等全部结束
4. **电路中段能力**——如果输出跑飞，流式可以在中间截断，阻塞做不到
5. **子 agent 流合并**——并行子 agent 的输出可以实时合并

### 后果

**正面：**
- 用户体验显著提升
- 电路中段成为可能
- 成本报告更精确（输出 token 实时累加）

**负面：**
- 核心循环复杂度增加（事件匹配替代简单函数调用）
- 成本预估算只能覆盖输入侧（输出成本只能在流结束后确认）
- CLI 渲染需要处理增量更新（不是所有终端都支持）
- 非交互式环境（CI/CD）需要 `--no-stream` 逃生口

## ADR-003: MCP 作为插件协议

**状态：** 已采纳 · **日期：** 2026-05

### 背景

Sponge 需要一个插件系统来扩展工具集。问题是：用自定义协议还是用行业标准？

### 考虑的方案

| 方案 | 描述 |
|------|------|
| **A: 自定义协议** | 定义自己的 Plugin ABC + JSON-RPC 接口。所有工具通过此接口。 |
| **B: MCP（选定）** | 实现 MCP 客户端，包装标准 MCP server。内置插件仍然本地实现。 |
| **C: 双协议** | 同时支持自定义协议和 MCP。 |

### 决策

选择 **B**。理由：

1. **生态**——2026 年 MCP 已经被 Claude Code、Cursor、Copilot 采用，有数百个现成 server
2. **零适配成本**——Sponge 用户可以立即使用 `@modelcontextprotocol/server-filesystem`、`server-github` 等
3. **内置插件本地实现**——`file_ops`、`shell` 等高频操作不走 MCP，零额外延迟
4. **双协议（方案 C）被拒绝**——因为每个第三方工具开发者只需要学一个协议

### 后果

**正面：**
- 开箱即用数百个工具
- 与生态兼容——不是另一个轮子
- 内置插件保留零开销路径

**负面：**
- MCP 的 stdio/SSE 跳转增加 ~1-5ms 延迟（只在非内置工具上）
- MCP 协议的功能受限于其工具模型（没有 sub-agent 概念）
- MCP server 是子进程——生命周期管理（崩溃、重启、资源限制）增加了复杂度
- 安全边界：MCP server 是外部代码，需要沙箱隔离

## ADR-004: 自调优用 SQL + 简单统计，不用 ML

**状态：** 已采纳 · **日期：** 2026-05

### 背景

自调优系统需要从遥测数据中发现模式并调整参数。实现路径有多种。

### 考虑的方案

| 方案 | 描述 |
|------|------|
| **A: 机器学习模型** | 训练一个轻量模型（线性回归 / 决策树）预测最优参数 |
| **B: SQL + 简单统计（选定）** | 纯 SQL 查询聚合遥测数据 + 统计测试（Mann-Whitney U）验证变更 |
| **C: 强化学习** | 把参数调优建模为 RL 问题，用 session 成本作为 reward |

### 决策

选择 **B**。理由：

1. **可解释性**——每个提案都有 SQL 查询和统计检验作为依据，可以审查
2. **零依赖**——不需要 scikit-learn、PyTorch 或任何 ML 框架
3. **零训练数据**——ML 方案需要标注数据，在冷启动期不可用
4. **足够好**——需要优化的模式很简单（请求间隔、压缩率、重复任务），模式明确，不需要复杂模型
5. **调试友好**——`sponge tune --report` 可以直接显示 SQL 查询结果

### 后果

**正面：**
- 实现简单（~500 行覆盖四个组件）
- 每行代码都可审计
- 没有 ML 特有的运维负担（模型版本管理、训练 pipeline、数据漂移）
- 响应式——不是预测式（检测到模式后才响应，不是预测未来的最优参数）

**负面：**
- 不能发现非线性模式（如"下午 2-4 点请求更密集但更短"这种时间+长度组合模式）
- 参数间交互效应被忽略（提高 masking_threshold 和降低 budget_ceiling 可能互相抵消）
- 需要更长时间才能收敛（ML 方案理论上可以用更少 session 找到更优参数）

## ADR-005: 三层记忆模型

**状态：** 已采纳 · **日期：** 2026-05

### 背景

Sponge 需要在会话之间保持信息——用户的偏好、项目约定——但不同粒度的信息需要不同的存储和生命周期。

### 考虑的方案

| 方案 | 描述 |
|------|------|
| **A: 单层（全局）** | 所有记忆存在 `~/.sponge/memory.toml`，所有项目共享 |
| **B: 双层（项目 + 全局）** | 项目级 `.sponge/memory.toml` + 全局 `~/.sponge/preferences.toml` |
| **C: 三层（选定）** | Session（对话历史）+ Project（项目规则）+ User（用户偏好） |

### 决策

选择 **C**。理由：

1. **职责分离清晰**——session 级别存"刚才说了什么"，project 级别存"这个项目的规矩"，user 级别存"我一直以来的习惯"。
2. **注入策略不同**——session 是消息历史（不注入 system prompt），project 和 user 都注入 system prompt。
3. **跨项目隔离**——Project A 的规则不会泄露到 Project B。
4. **与生态系统一致**——Claude Code 有 CLAUDE.md（≈ project memory），Cursor 有 .cursorrules（≈ project memory），但两者都没有 user 级别的全局偏好存储。

### 后果

**正面：**
- 清晰的作用域和优先级（Session < Project < User）
- 系统 prompt 只看到相关规则（不会把项目 A 的规则注入到项目 B 的会话）
- 用户可以设置全局默认值（模型选择、budget ceiling）而不用每个项目重复

**负面：**
- 三层之间的冲突需要解决规则（Project > User，但什么是"冲突"需要定义）
- 每个 session 启动时需要读取和合并三个源，增加 ~2-5ms 启动开销
- 用户需要理解三层模型才能有效管理记忆

## ADR-006: 三梯级审批门

**状态：** 已采纳 · **日期：** 2026-05

### 背景

零成本插件（file_ops、shell）安全但危险。需要在不打断用户工作流的前提下防止误操作。

### 考虑的方案

| 方案 | 描述 |
|------|------|
| **A: 全自动** | 所有工具自动执行，没有审批 |
| **B: 全确认** | 每个工具执行前都弹窗确认 |
| **C: 二梯级（Allow / Reject）** | 用户预设每个工具是"允许"还是"拒绝" |
| **D: 三梯级（选定）** | Allow（自动执行）+ Confirm（弹窗确认）+ Reject（阻止） |

### 决策

选择 **D**。理由：

1. **全自动（A）**——`rm -rf /` 是一行 hallucination 的距离。不可接受。
2. **全确认（B）**——每个 `file_read` 都要点确认 → 用户学会盲按 Y → 失去保护作用。不可接受。
3. **二梯级（C）**——静态预设无法覆盖所有场景。"写文件"有时是安全的（在 tmp 目录），有时是危险的（覆盖生产配置）。
4. **三梯级（D）**——Allow 覆盖高频安全操作（读文件、搜索），Confirm 覆盖中风险操作（写文件、执行命令），Reject 覆盖永不安全操作（rm -rf）。最平衡。

### 后果

**正面：**
- 安全（Reject 阻止最坏情况）
- 可用（Allow 不打断低频安全操作）
- 灵活（Confirm 在上下文中做决定）

**负面：**
- 预设分级的准确性取决于配置——如果用户把所有工具都设为 Allow，保护失效
- Confirm 时的弹窗 UX 需要精心设计（显示 diff/命令预览、接受 Y/n/o/a 等多种响应）
- `--auto-approve` 逃生口可能被滥用（但在 CI 场景下是必需的）

## ADR-007: Python 作为实现语言

**状态：** 已采纳 · **日期：** 2026-05

### 背景

Agent harness 可以用多种语言实现。关键考量：生态、性能、贡献门槛。

### 考虑的方案

| 方案 | 优势 | 劣势 |
|------|------|------|
| **Python** | AI 生态母语、直接绑定 tiktoken/SQLite、asyncio、低贡献门槛 | GIL、部署复杂 |
| **TypeScript** | Node 生态、VSCode 集成好、原生事件循环 | 缺少 tiktoken 级 tokenizer 绑定 |
| **Rust** | 极致性能、无 GC pause、单二进制分发 | 开发速度慢、AI 生态差 |
| **Go** | 简单部署、静态二进制、好并发 | 泛型弱、AI SDK 生态差 |

### 决策

选择 **Python**。决定性因素：

1. **生态是硬约束**——需要直接调用 anthropic、openai SDK，这俩都是 Python 原生。TypeScript 版是社区维护的二级公民。
2. **tiktoken**——Python 版是官方的，其他语言的绑定可能滞后。
3. **LLM 是瓶颈**——95% 的延迟花在 API 调用上，不是语言运行时。Python 的性能足够。
4. **贡献门槛**——Sponge 的目标用户是 AI 开发者，Python 是他们的 lingua franca。Rust 会大幅缩小贡献者池。

### 后果

**正面：**
- 直接使用所有 Python AI SDK（Anthropic、OpenAI、tiktoken）
- 最大潜在贡献者池
- async/await 对 IO 密集型工作负载足够好

**负面：**
- 部署需要 Python 运行时（不能单二进制分发）
- GIL 限制 CPU 密集型操作（但这类操作在 Sponge 架构中极少——只有 token counting 是 CPU 密集的）
- 依赖管理（pip、虚拟环境）增加了用户摩擦

## ADR-008: SQLite 作为遥测存储

**状态：** 已采纳 · **日期：** 2026-05

### 背景

自调优系统需要持久化遥测数据。存储方案的选择影响可靠性、运维复杂度和隐私。

### 考虑的方案

| 方案 | 优势 | 劣势 |
|------|------|------|
| **SQLite** | 零运维、零服务器、ACID、SQL 查询 | 写并发有限（但对单用户场景足够） |
| **PostgreSQL** | 高并发、高级查询 | 需要服务器、运维负担、违反"本地优先" |
| **TimescaleDB** | 时序数据优化 | 需要 PostgreSQL、过度设计 |
| **JSON Lines 文件** | 极致简单 | 没有索引、查询复杂度 O(n)、无一致性保证 |
| **DuckDB** | 分析查询快 | 写入性能差、生态新 |

### 决策

选择 **SQLite**。理由：

1. **本地优先，隐私架构**——没有数据离开机器。这是设计约束，不是可选特性。
2. **零运维**——`sqlite3.connect()` 一行代码。不需要安装、配置、备份。
3. **ACID**——崩溃安全。如果 session 在写入中间崩溃，SQLite 自动恢复。
4. **SQL 查询**——"请求间隔中位数"、"压缩比分布"等分析模式直接用 SQL 表达，不需要 ORM。
5. **~20 bytes/event 的写入开销**——一个 session 产生 < 100 事件，全年的数据量 < 10MB。

### 后果

**正面：**
- 完全离线、完全私密
- 零运维成本
- 分析查询直接用 SQL

**负面：**
- 写并发有限——多个 session 同时写可能冲突（但单用户场景几乎不会发生）
- 没有内置时间序列功能——需要 `datetime('now', '-7 days')` 手动做时间窗口
- SQLite 的 JSON 查询能力有限——复杂嵌套的 JSON data 字段查询不便（但设计上限制了 data 为浅层 JSON）

## ADR-009: 五层上下文压缩管线

**状态：** 已采纳 · **日期：** 2026-05

### 背景

上下文压缩是一个"how far do you go"的问题。压得太少浪费 token，压得太多丢失信号。

### 考虑的方案

| 方案 | 描述 | 问题 |
|------|------|------|
| **单层（滑动窗口）** | 只保留最后 N 轮对话 | 粗暴、丢失可能重要的历史 |
| **双层（屏蔽 + 滑动窗口）** | 屏蔽旧工具输出 + 窗口 | 压缩率有限 |
| **三层（屏蔽 + 修剪 + 窗口）** | 增加基于重要性的消息选择 | 仍不够：超出窗口的历史全部丢失 |
| **四层 + 总结 + 窗口** | 增加 LLM 总结作为最后手段 | 总结有成本 |
| **五层 + 服务端清除 + ...（选定）** | 以上全部 + Anthropic 服务端 API 利用 | 最复杂但压缩率最高 |

### 决策

选择 **五层管线**。为什么不是更少？

每一层解决一个独立的问题：
- **L1（服务端清除）**——对使用者零成本，provider 独有功能
- **L2（屏蔽）**——最适合工具输出（体积大、引用少）
- **L3（修剪）**——恢复"被屏蔽覆盖不到"的空间（旧但重要的 turn）
- **L4（总结）**——唯一能将历史压缩到固定小体积（<300 tokens）的方法
- **L5（窗口）**——硬安全保障，确保不超过 context window

### 后果

**正面：**
- 压缩效果可分层记录，便于证明某个 workload 的实际 token reduction
- 每层独立可配置、可跳过
- L5 确保永远不会超过 context window

**负面：**
- 五层管线在每次 LLM 调用前运行，增加了 ~5-20ms 延迟（但相对于 LLM 调用的 1-10s，这可以忽略）
- L4（总结）本身有 token 成本——使用 DeepSeek V4-Flash 约 $0.0005/次
- L1 只对 Anthropic 有效——切换 provider 时自动变为 no-op（见 ADR-010）

## ADR-010: Provider 能力声明 + 自适应管线

**状态：** 已采纳 · **日期：** 2026-05

### 背景

不同的 LLM provider 有不同的 API 特性和成本结构。Anthropic 支持服务端 tool result clearing 和 10% 缓存读成本，OpenAI 是 50% 缓存读成本，DeepSeek 不支持图片。Sponge 需要适配这些差异。

### 考虑的方案

| 方案 | 描述 |
|------|------|
| **A: 最小公分母** | 只用所有 provider 都支持的特性。Layer 1 不存在，缓存策略固定。 |
| **B: Provider 特定分支** | `if provider == "anthropic": ... elif provider == "openai": ...` |
| **C: 能力声明（选定）** | 每个 provider 声明其能力，管线查询能力集后自适应 |

### 决策

选择 **C**。理由：

1. **最小公分母（A）**——浪费了 Anthropic 独有的省钱特性。不可接受。
2. **Provider 特定分支（B）**——每个新 provider 需要改管线代码。不可维护。
3. **能力声明（C）**——provider 只声明"我能做什么"，管线自动适配。新 provider 不需要改管线。

### 后果

**正面：**
- 新 provider 只需要实现能力声明，管线自动适配
- 用户可以随时切换 provider，不需要担心"哪层还活着"
- 每层的实际效果可查询：`sponge config show capabilities`

**负面：**
- 能力声明的 API 需要精心设计——既要足够表达力，又不能太复杂
- 某些层的"变体行为"（如不同的缓存定价）不是简单的开/关，需要数值参数
- 如果 provider 的能力在未来版本中变化，能力声明也需要跟着更新
